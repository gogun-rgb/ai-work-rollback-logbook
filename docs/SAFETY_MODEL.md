# Safety Model

이 문서는 AI 작업 되돌리기 로그북이 파일 복구를 안전하게 처리하기 위해 사용하는 원칙과 제한을 설명합니다.

## Git 실행 방식

Git 명령은 shell 문자열로 조합하지 않고 `execFile`과 인자 배열로 실행합니다. 이 방식은 파일 경로가 옵션이나 shell 문법으로 다시 해석되는 위험을 줄입니다.

## 프로젝트 및 경로 차단

`assertSafeRelativePath`는 다음 경로를 거부합니다.

- 절대 경로
- `..`로 프로젝트 밖을 가리키는 경로
- `path.resolve` 이후 프로젝트 루트 밖으로 벗어나는 경로

복구와 diff 조회는 프로젝트 내부 상대 경로만 처리합니다.

## 상태 서명 구성

파일별 상태 서명은 사용자가 검토한 시점의 상태를 복구 직전에 다시 확인하기 위한 해시입니다. 서명에는 다음 정보가 포함됩니다.

- 파일 경로
- 이전 파일 경로
- porcelain Git 상태
- 앱에서 분류한 Git 상태
- 추가 및 삭제 줄 수
- Git 인덱스 상태
- 작업트리 blob 해시
- staged diff 해시
- unstaged diff 해시

줄 수가 같아도 내용이 바뀌면 작업트리 blob 또는 diff 해시가 바뀌므로 복구가 차단됩니다.

## 복구 직전 재검증

복구 요청이 들어오면 저장된 서명을 바로 신뢰하지 않습니다. 현재 Git 상태를 다시 스캔하고, 요청된 파일의 현재 서명과 사용자가 검토했던 서명을 비교합니다. 하나라도 다르면 전체 복구 요청을 중단합니다.

## staged/worktree 동시 복구

지원되는 추적 파일은 다음 명령으로 복구합니다.

```bash
git restore --staged --worktree -- <file>
```

따라서 staged-only 수정도 인덱스와 작업트리 양쪽에서 되돌릴 수 있습니다.

## untracked 파일 미삭제 원칙

추적되지 않은 파일은 자동 삭제하지 않습니다. `git clean` 계열 명령은 사용하지 않으며, untracked 파일은 사용자가 직접 확인하고 삭제해야 합니다.

## 지원하지 않는 Git 상태

MVP에서 자동 복구하지 않는 상태는 다음과 같습니다.

- 새로 staged된 파일
- 이름이 변경된 파일
- 추적되지 않은 새 파일
- 원격 저장소 상태 또는 Pull Request 관련 상태

이 상태들은 목록과 diff 확인 대상일 수 있지만 복구 단계에서는 건너뛰고 사용자에게 제한을 안내합니다.

## 알려진 위험과 제한

- 복구 전 백업 파일을 자동 생성하지 않습니다.
- 여러 프로젝트를 동시에 비교하지 않습니다.
- Git submodule, merge conflict, sparse checkout 같은 고급 상태를 별도로 모델링하지 않습니다.
- 바이너리 파일은 diff 본문 대신 바이너리 안내로 표시합니다.
- 대용량 diff는 UI 보호를 위해 일부 줄만 표시합니다.
